# 创作者画像与选题指标落地设计

## 1. 目标

把创作者的账号资料和最近作品元数据转换为稳定的结构化画像，再把画像用于热点领域匹配、内容形式匹配、创作者体量解释和风险边界判断。

画像生成由大模型完成。程序不根据标题关键词直接宣布“这是某类创作者”，程序只负责：限制最多 50 条输入、构造提示词、接收 JSON、校验字段、规范化空值，并把解析结果交给指标模块。

## 2. 输入

输入分为两类：

1. `explicit_profile`：用户或平台明确提供的账号信息，例如创作者名称、平台、粉丝数、目标受众、账号自述、目标平台和风险偏好。
2. `history`：最多 50 条最近作品元数据。每条优先提供 `title`、`description`、`tags`、`published_at`、`duration_seconds`、`view_count`、`url`。不需要把正文和评论全文传给画像模型。

缺少粉丝数时，模型必须输出 `creator_size.level=unknown`，不能用播放量冒充粉丝量。缺少时长时，长短视频形式为 `unknown`，不能仅凭标题猜测。

## 3. LLM 调用

画像阶段使用独立 system prompt，不开放搜索、数据库或业务工具。输入采用 Chat Completions：

```json
{
  "messages": [
    {"role": "system", "content": "你是创作者画像归纳器...只输出一个 JSON 对象"},
    {"role": "user", "content": "{\"recent_work_samples\":[...]}"}
  ],
  "response_format": {"type": "json_object"},
  "stream": true
}
```

模型的任务是归纳，不是推荐选题。要求：只根据证据；证据不足写 `unknown`；身份标签短而稳定；不要把单个标题直接当作账号定位；只输出 JSON，不输出 Markdown 或解释文本。

## 4. 输出画像

模型输出必须包含：

- `creator_identity_tags`：用于检索和匹配的身份标签，如“科技区”“教程型”。
- `content_positioning`：一句话定位、价值主张、表达风格。
- `content_format`：`short_video`、`long_video`、`mixed` 或 `unknown`，以及时长区间。
- `creator_size`：`top`、`large`、`medium`、`small`、`nano` 或 `unknown`，同时记录粉丝数和播放中位数。
- `domain_profile`：主领域、次领域、主标签、避开标签。
- `audience_profile`：受众描述、受众需求、知识水平。
- `evidence_summary`：样本条数、支持的结论、未知项。
- `confidence`：`high`、`medium` 或 `low`。

程序解析后会映射为下游统一字段：`domain`、`main_tags`、`avoid_tags`、`content_positioning`、`content_format`、`creator_size`、`target_platforms`、`history_summary`。指标模块只使用这个解析后的对象。

## 5. 解析与失败处理

解析器先去除偶发的 JSON 代码围栏，再读取第一个 JSON 对象；检查必需对象和枚举字段；数组去重并截断；数值转为非负整数；无法确定的数字转为 `null`。

LLM 输出无法解析时，工具返回错误，不把自然语言错误当成画像。只有调用方显式传入 `allow_rule_fallback=true` 时，才允许使用规则画像，并在 `generation.method` 标记为 `rule_fallback`。生产默认关闭兜底，避免规则画像伪装成模型结论。

## 6. 与后续选题流程的连接

画像完成后：

1. `hot_news_fetcher` 从数据库提取最多 50 条热点，`web_search_fetcher` 补充联网证据。
2. 归一化和聚类得到事件簇。
3. 领域匹配读取 `domain`、`main_tags`、`avoid_tags`。
4. 形式匹配读取 `content_format` 和 `target_platforms`。
5. 体量只作为解释上下文，不直接提高或降低热点分。
6. 证据和风险独立计算，画像模型没有提供的事实不得补入最终推荐。

## 7. 工具落地

- `creator_profile_builder`：独立 LLM 调用和 JSON 解析。
- `hot_news_fetcher`：固定只读 SQL，最多 50 条。
- `proposal_topic_value_scorer`：单事件或批量事件的 proposal_v2 评分；批量调用一次 LLM 生成语义匹配分，再计算选题价值。
- `content_angle_planner`：最多五个创作角度，不产生事实断言。
- `risk_evidence_checker`：风险折扣、来源数量和待补证据。
- `creator_topic_recommender`：把画像、事件和证据连接起来排序。

推荐聚合器要求传入已经解析的 `creator_profile`，不在缺少画像时偷偷使用规则画像；这样能保证“画像由大模型生成”的业务约束可追踪。
